# 제품 요구사항

## 문제

현재 작업자는 정형화된 Word 파일을 하나씩 분석하고, LLM에 HTML 템플릿과 함께 전달해 변환한 뒤, 다시 전체 문서를 합치는 반복 작업을 수행합니다. 파일 수가 늘수록 복사·붙여넣기, 누락 검사, 스타일 수정 비용이 증가합니다.

## 목표

- 한 번의 작업으로 여러 Word 문서를 변환한다.
- 파일마다 동일한 규칙의 개별 HTML을 생성한다.
- 전체 내용을 지정된 순서로 합친 통합 HTML을 추가 생성한다.
- 출력 HTML의 스타일은 inline CSS여야 한다.
- 사용자가 개발자 도움 없이 템플릿을 만들고 재사용할 수 있어야 한다.

## MVP 사용자 여정

1. 사용자가 템플릿 이름, 개별 템플릿, 통합 템플릿을 등록한다.
2. Word 파일 여러 개를 선택하고 순서를 정한다.
3. 변환 전 예상 필드와 경고를 확인한다.
4. 변환을 실행한다.
5. 파일별 결과를 미리보고 필요한 항목만 다시 생성한다.
6. 개별 HTML, 통합 HTML, 이미지, 리포트가 담긴 ZIP을 받는다.

## 템플릿 모델

템플릿은 두 종류를 한 세트로 관리합니다.

- `document_template`: Word 파일 하나를 렌더링
- `combined_template`: 렌더링된 문서 목록과 공통 메타데이터를 렌더링

사용자 템플릿은 HTML과 inline CSS를 직접 편집할 수 있으며, 시각 편집기는 후속 기능으로 둡니다. MVP는 코드 편집기, 슬롯 자동완성, 실시간 미리보기에 집중합니다.

## 범위 제외(MVP)

- `.doc` 구형 바이너리 형식 직접 지원
- Word 레이아웃의 픽셀 단위 복제
- 다중 사용자 권한 및 결제
- 자유 형식 문서의 완전 자동 의미 해석
- Office macro 실행

## 성공 지표

- 10개 문서 작업의 수동 개입 시간을 기존 대비 70% 이상 단축
- 지원 요소 기준 변환 누락 1% 미만
- 같은 입력/설정의 반복 변환 결과가 구조적으로 동일
- 사용자가 오류 문서를 식별하고 개별 재처리 가능

## 확정에 필요한 예시

- 실제 구조를 대표하는 Word 파일 2~3개
- 현재 개별 HTML 템플릿 1개 이상
- 통합 HTML의 현재 예시 또는 원하는 목업
- 이미지 처리 방식: HTML 내 base64, 별도 assets, 또는 외부 URL
- 최종 HTML 사용처: 이메일, CMS, 웹 페이지, 다른 시스템 업로드
