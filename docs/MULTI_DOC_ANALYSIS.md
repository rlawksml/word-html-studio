# 2026년 7월 Word 4개 비교 분석

## 제품 관점의 결론

이 업무는 “Word 디자인을 HTML로 복제”하는 작업이 아니라, 여러 작성자가 보낸 반정형 Word에서 의미 구조를 추출하고 공통 HTML 디자인으로 편집하는 작업입니다.

사용자나 작성자는 JSON과 HTML을 작성하지 않습니다. 기존 Word 파일을 그대로 제출할 수 있으며, 프로그램이 내부 구조를 만든 뒤 사용자가 미리보기에서 교정합니다.

## 문서별 규모

| 문서 | 문단 | 이미지 | 표 | 사용 스타일 | 특징 |
|---|---:|---:|---:|---|---|
| 수연목서 | 22 | 1 | 0 | Normal, List Paragraph | 전시 소식 1개 |
| 소담쓰담 | 33 | 3 | 0 | Normal, List Paragraph | 소식 3개 |
| 오직 책방 | 27 | 2 | 0 | Normal, List Paragraph | 긴 프로그램 소식 1개 |
| 책빵 자크르 | 181 | 14 | 0 | Normal, List Paragraph | 소식이 많고 중첩 목록이 복잡함 |

네 파일 모두 Word의 `제목`, `제목 1`, `제목 2` 스타일을 사용하지 않습니다. 이미지도 inline shape가 아니라 floating drawing으로 저장되어 있으므로 일반적인 `python-docx` 문단 순회만으로는 이미지 위치를 잃을 수 있습니다.

## 안정적으로 공통인 구조

모든 문서는 다음 순서를 거의 동일하게 사용합니다.

1. `[지관서가 동네 책방 YYYY년 M월 소식 – 책방명]`
2. 책방 이름
3. 주소, 영업시간, 연락처, SNS 등의 기본 정보 목록
4. `동네 책방 ... 소식 안내` 소개 문장
5. 소식 제목
6. 이미지와 `항목명: 내용` 상세 정보

이 공통 골격은 규칙 기반 파서로 처리할 수 있습니다. LLM이 없어도 제목, 책방 정보, 소개 문장과 대부분의 필드를 추출할 수 있습니다.

## 문서마다 다른 부분

### 소식 제목 표기

- `<소식 제목>`
- `<소식 제목> [신청마감]`
- `<AFTER USE> 김우영 작가 사진전`
- `박조건형 작가와 함께 <자기내면 탐구 글쓰기 놀이>`
- 번호가 붙은 제목 또는 꺾쇠가 없는 제목

따라서 꺾쇠만으로 소식 경계를 나누면 누락이 발생합니다. 문단 강조, 앞뒤 빈 문단, 이미지 위치, 뒤따르는 `일시:`·`내용:` 같은 필드를 함께 봐야 합니다.

### 상세 필드

공통 필드는 있지만 문서별 확장 필드도 많습니다.

- 공통: 내용, 일시/일정, 장소, 참가비, 신청 방법, 문의
- 전시: 전시 소개, 기간
- 독서모임: 선정도서, 진행 방식
- 모집: 모집기간, 모집대상, 활동기간, 활동방식
- 캠페인: 주관, 주최, 지원, 후원, 상품

중간 JSON은 필드를 고정된 열로 제한하지 않고 `fields[]` 배열로 보존해야 합니다. 자주 쓰는 필드만 표준 키로 정규화합니다.

### 분량

짧은 문서는 소식이 1개이지만 자크르는 다수의 소식과 181개 문단을 포함합니다. 개별 HTML에는 전체 내용을 담을 수 있어도 통합 HTML에는 책방당 핵심 소식 1~3개만 선택해야 합니다.

## 권장 변환 구조

```text
Word 파일
  → OOXML에서 문단·목록·링크·floating 이미지 순서 추출
  → 규칙 기반으로 문서 헤더와 기본 정보 분리
  → 소식 후보 경계 탐지
  → 필드명 정규화
  → 낮은 신뢰도 블록만 LLM에 분류 요청
  → 사용자가 소식 경계와 이미지 귀속 검토
  → 개별 HTML 렌더링
  → 핵심 소식 선택/요약 검토
  → 통합 HTML 렌더링
```

## 작성자에게 요구할 최소 규칙

기존 작성 습관을 크게 바꾸지 않고 다음 다섯 가지만 안내하는 것이 현실적입니다.

1. 첫 줄은 `[지관서가 동네 책방 YYYY년 M월 소식 – 책방명]`으로 작성
2. 주소·영업시간·연락처·SNS는 `항목명: 내용`으로 작성
3. 새로운 소식은 새 문단에서 시작하고 가능하면 `<소식 제목>`으로 표시
4. 소식 상세도 `내용:`, `일시:`, `장소:`, `신청 방법:`처럼 항목명을 붙임
5. 이미지는 관련 소식 제목과 상세 내용 사이 또는 바로 뒤에 배치

Word 제목 스타일은 정확도를 높이는 권장 사항으로 두되 필수로 강제하지 않습니다.

## 프로그램 검토 화면에서 반드시 보여줄 것

- 감지한 책방명, 연월, 주소, 연락처
- 감지한 소식 개수와 각 소식 제목
- 각 이미지가 어느 소식에 연결되었는지
- 날짜, 모집 상태, 신청 링크
- 통합본에 포함할 핵심 소식 선택
- 원문과 자동 요약의 나란한 비교
- 신뢰도가 낮거나 누락 가능성이 있는 항목

## LLM 사용 경계

규칙 기반으로 처리할 항목:

- 문서 제목과 책방명
- 기본 정보의 `항목명: 값` 분리
- URL, 이메일, 전화번호와 날짜 추출
- 원본 이미지 추출
- 템플릿 렌더링과 inline CSS

LLM이 보조할 항목:

- 불규칙한 소식 제목과 경계 판별
- 긴 내용을 통합본용 1~2문장으로 요약
- 상태 배지 후보 생성: 모집 중, 신청 마감, 진행 예정, 종료
- 이모지와 짧은 홍보 문구 제안

LLM 결과는 원문을 대체하지 않고 제안으로 저장합니다. 날짜나 모집 상태처럼 사실성이 중요한 값은 원문 근거와 함께 사용자 확인을 받아야 합니다.

## 개인정보와 외부 전송

문서에는 전화번호, 이메일, 계좌번호가 포함될 수 있습니다. 외부 LLM을 사용할 때는 전체 Word를 보내지 말고 필요한 텍스트 블록만 전송해야 합니다. 계좌번호 등 요약에 불필요한 값은 기본적으로 마스킹하거나 제외하고, 전송 범위를 작업 화면에 표시합니다.
