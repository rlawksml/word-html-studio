# 동네책방 소식 스튜디오 — 개발 에이전트 지침

이 파일은 저장소 전체에 적용됩니다. 앞으로 이 프로젝트를 수정하는 에이전트는 기능 구현 전에 이 규칙을 확인하고, 기존 구조와 사용자 흐름을 보존해야 합니다.

## 1. 기본 원칙

- 이 서비스는 개발자가 아닌 일반 사용자의 월간 소식 입력 업무를 단순하게 만드는 제품이다. 새 기능 때문에 입력 단계가 불필요하게 늘어나지 않게 한다.
- 기존 코드와 사용자 변경을 보존하고, 요청과 무관한 대규모 재작성은 하지 않는다.
- Atomic Design을 기본 구조로 사용하되 작은 UI까지 억지로 atom으로 만들지 않는다. 재사용성보다 책임 분리가 더 명확한 구조가 있으면 기능 단위 분리를 선택할 수 있다.
- 하나의 파일에 화면, 상태, API 호출, 데이터 변환, 스타일 책임을 모두 모으지 않는다.
- 기능을 추가할 때 구현뿐 아니라 관련 테스트, README, TODO와 설계 문서의 상태도 함께 확인한다.

## 2. Next.js와 UI 구조

현재 계층의 책임을 다음과 같이 유지한다.

```text
app/                 라우트 진입점, layout, Route Handler
components/atoms/    더 나눌 필요가 없는 작은 공통 UI
components/molecules/하나의 명확한 상호작용 또는 정보 단위
components/organisms/역할별 주요 작업 화면과 큰 기능 영역
components/templates/페이지 구성과 역할별 화면 선택
hooks/               화면 상태, 사용자 흐름, 재사용 가능한 브라우저 동작
lib/                 HTML 생성, API 클라이언트, 타입, 포맷터와 순수 로직
styles/              공통 기반 및 역할·기능별 스타일
```

- `app/page.tsx`와 `app/layout.tsx`는 조합 지점으로 유지한다. 업무 로직이나 큰 JSX를 다시 넣지 않는다.
- 컴포넌트는 한 가지 사용자 목적을 설명할 수 있어야 한다. 서로 다른 역할의 화면이나 세 개 이상의 독립된 섹션이 섞이면 분리를 검토한다.
- API 요청과 Supabase 호출을 표현 컴포넌트에 직접 넣지 않는다. 브라우저 요청은 `lib/*-client`, 서버 저장소 호출은 Route Handler와 서버 모듈 경계를 사용한다.
- 여러 화면이 공유하는 상태와 명령은 controller hook에 둔다. 한 화면에서만 쓰는 열림/닫힘 같은 UI 상태는 해당 컴포넌트 가까이에 둔다.
- controller hook도 하나의 거대한 파일로 방치하지 않는다. 저장, 이미지, 발행, 방문자 탐색처럼 독립 흐름이 커지면 작은 hook 또는 use-case 함수로 분리한다.
- 파일이 약 200~300줄을 넘는 것만으로 기계적으로 분리하지는 않지만, 두 가지 이상의 책임이 있거나 변경 이유가 여러 개라면 먼저 분리한다.
- 이름만 감싸는 컴포넌트와 한 번만 쓰이는 무의미한 추상화는 만들지 않는다.

## 3. CSS 분리 규칙

- `app/globals.css`에는 스타일 구현을 쌓지 않는다. 전역 import 순서와 정말 전역인 선언만 둔다.
- 공통 토큰, reset, typography, 버튼·폼 같은 기반 스타일은 `styles/foundations.css`에 둔다.
- 방문자, 정보 입력자, HTML 편집자처럼 역할이 다른 화면은 각각의 역할별 CSS 파일을 유지한다.
- 독립 기능의 스타일이 커지면 역할별 파일에 계속 누적하지 말고 기능 이름의 CSS 파일로 분리한다. 예: `photo-group.css`, `preview-dialog.css`.
- 한 CSS 파일에 서로 관련 없는 화면 영역이 세 개 이상 포함되거나 약 300줄을 넘기 시작하면 분리를 검토한다.
- 두 곳 이상에서 실제로 같은 형태로 사용되는 규칙만 공통화한다. 앞으로 쓸 것 같다는 이유만으로 미리 공통 클래스를 만들지 않는다.
- 사이트 UI는 class 기반 CSS를 사용한다. inline CSS는 외부 게시판에 복사하는 HTML 생성 결과처럼 제품 요구상 필요한 경우에만 사용한다.
- 역할별 CSS import 순서가 cascade에 영향을 주므로 `globals.css`의 import 순서를 변경할 때는 회귀 테스트를 수행한다.
- 모바일 규칙은 관련 기본 스타일과 함께 설계하되, 공통 breakpoint 조정은 `styles/responsive.css`에 둔다. 특정 기능의 모바일 규칙이 많아지면 기능 CSS 내부의 media query로 옮길 수 있다.

## 4. 기능 추가 시 분리 판단

새 기능을 구현하기 전에 다음 순서로 판단한다.

1. 기존 atom, molecule, organism 또는 hook의 책임 안에 들어가는가?
2. 기존 파일에 넣으면 그 파일의 변경 이유가 두 개 이상이 되는가?
3. 다른 역할이나 화면에서도 재사용되는가?
4. 상태, API, 표시 로직을 서로 독립적으로 테스트할 수 있는가?
5. CSS가 공통 기반인지, 역할 전용인지, 기능 전용인지 구분됐는가?

새로운 독립 상호작용이면 molecule 또는 전용 hook을 우선 검토한다. 역할 전체의 흐름이면 organism, 여러 역할에 영향을 주는 업무 명령이면 controller/use-case 경계를 사용한다.

## 5. 데이터와 보안 경계

- Supabase Secret Key와 작업 암호는 서버 환경변수에서만 사용하고 브라우저 번들, 로그, 테스트 fixture, GitHub에 포함하지 않는다.
- 원본 사진은 비공개, 모바일 미리보기는 공개라는 현재 저장 원칙을 유지한다.
- 기존 JSON 데이터에 필드를 추가할 때는 선택 필드와 기본값으로 이전 데이터 호환성을 먼저 확보한다.
- 방문자는 읽기만 가능해야 하며 입력자와 HTML 편집자의 권한 경계를 약화하지 않는다.
- 외부 HTML 생성 값은 escape 및 URL 검증을 거치며, 생성 HTML의 inline CSS와 애플리케이션 UI CSS를 섞지 않는다.

## 6. 검증과 문서

- 변경 후 범위에 맞게 `npm run lint`, `npm test`, `npm run build`, `git diff --check`를 실행한다.
- UI 구조를 이동하면 회귀 테스트의 파일 목록과 책임 설명도 함께 수정한다.
- 구현 상태가 바뀌면 `README.md`, `TODO.md`, `docs/ARCHITECTURE.md` 중 영향받는 문서를 갱신한다.
- GitHub에는 기능 단위 `codex/<milestone>-<feature>` 브랜치와 PR을 사용한다. PR에는 작업 내용, 검증, 문서 변경, 민감정보 확인과 남은 위험을 기록한다.

## 7. 완료 전 체크리스트

- route와 template가 다시 비대해지지 않았는가?
- 컴포넌트가 API와 저장소 구현을 직접 알지 않는가?
- 새 CSS를 `globals.css`나 한 역할 파일에 무조건 누적하지 않았는가?
- 공통 스타일과 기능 전용 스타일의 위치가 명확한가?
- 데스크톱과 모바일 사용자 흐름을 모두 고려했는가?
- 기존 데이터, 기존 HTML 생성 결과와 호환되는가?
- 비밀값과 실제 사용자 원본 데이터가 커밋 대상에서 제외됐는가?
